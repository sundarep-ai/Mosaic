## Bucket 01 — Frontend: Add/Edit Expense Flow & Split Legibility

Source: `review_order/01-frontend-add-expense-flow.md`

### 1. Settle prefills the amount and direction (A1.1)
**Status: Done**

- `frontend/src/pages/Landing.jsx` — the "Settle" link now computes the actual debtor/creditor from `balanceAmount` (backend convention: positive = userA owes userB) and passes `amount`, `paid_by` (the debtor), and `split_method: "100% {creditor}"` in the navigation state. When the balance is already settled (`< $0.01`), it falls back to the old bare `description`/`category` state so there's nothing nonsensical to prefill.
- `frontend/src/pages/AddExpense.jsx` — the initial form state now consumes `prefill.amount` and `prefill.paid_by` directly (previously only `description`/`category`/a hardcoded `"100% other"` split were read, amount was always blank, and payer always defaulted to whoever is logged in).
- **Verify manually:** settle from both directions (you owe / partner owes) in shared/blended mode and confirm the amount and payer prefill correctly each time.

### 2. Split preview at the moment of commitment (A1.2)
**Status: Done**

- New `frontend/src/utils/splitPreview.js` — pure function `getSplitPreview(expense, me, other, fmt)` that reuses `calcMyPortion` (from `utils/portion.js`) to compute the net effect of the current amount/split/payer on the shared balance, and returns a sentence like `"50/50 → you'll owe Bob $12.50."` or `"100% → Bob will owe you $100.00."`. Returns `null` for zero/empty amounts, and for `category === "Payment"` (settle-up screens already state their own direction explicitly, so a second, differently-derived sentence there would be redundant/confusing rather than clarifying).
- `frontend/src/pages/AddExpense.jsx` — renders this sentence directly under the hero amount input whenever not in personal mode, recomputed live off `amount`, `split_method`, `paid_by`, and `category`.
- Covered by `frontend/src/utils/splitPreview.test.js` (50/50 both directions, 100%-owed both directions, Personal both payers, Payment category, zero amount).

### 3. "Save & add another" + don't wipe half-typed entries (A1.4)
**Status: Done**

- `frontend/src/pages/AddExpense.jsx` — added a secondary "Save & add another" button (create flow only). On success it resets `description`, `amount`, and `category` back to defaults but **keeps** `date`, `paid_by`, and `split_method` (the fields that are usually the same across a batch of receipts), shows a brief success banner, and refocuses the amount field instead of navigating away.
- `frontend/src/pages/AddNew.jsx` — both the Expense and Income forms are now always mounted; switching tabs toggles visibility with a `hidden` class instead of conditionally rendering one or the other. A half-typed entry in either tab now survives switching to the other tab and back.

### 4. Session expiry must not eat in-progress entries (A2)
**Status: Done**

- `frontend/src/pages/AddExpense.jsx` — immediately before every submit attempt (create or edit), the current form is written to `sessionStorage` (`mosaic_pending_expense_v1`). On success it's cleared immediately. On failure, it's only kept if the failure was specifically the "Session expired. Please log in again." error thrown by `fetchWithAuth` on a 401 — any other failure (validation, network) clears it, since the user still has their data live in the form and can just retry.
- On mount, the component checks `sessionStorage` for a stash matching the current context (same expense `id` if editing, or "new expense" if creating) and — if found — shows a banner offering **Restore** or **Discard**, rather than silently reapplying it. Submitting the form again (with or without restoring) clears any stale restore offer so it can't later overwrite a fresh in-progress entry.
- This directly addresses the flow in `fetchWithAuth.js` (global logout on 401) + `App.jsx` (renders `<Login/>` in place at the same URL, so `AddExpense` remounts fresh at the same route right after re-login) — the remount is exactly where the restore-check runs.
- **Verify manually:** start filling out an expense, expire the session (e.g. clear the auth cookie or wait out the 8h window), submit, log back in, and confirm the restore banner appears with the original values.

### 5. Surface backend validation messages for expenses (A2)
**Status: Done**

- `frontend/src/api/expenses.js` — `createExpense` and `updateExpense` now parse `err.detail` from the response body on failure and throw it as the error message, mirroring the existing pattern in `frontend/src/api/income.js`. Previously both threw a fixed generic string, so e.g. the backend's "Split method cannot assign 100% to the person who paid." (or any other 4xx validation message) never reached the user.
- `frontend/src/pages/AddExpense.jsx`'s catch block now surfaces `err.message` when present, falling back to the old generic copy only when there's no message.
- Covered by `frontend/src/api/expenses.test.js` (detail surfaced on 4xx, generic fallback on a bodyless 5xx, for both functions).

### 6. `inputMode="decimal"` on amount inputs (A1.7)
**Status: Done**

- `frontend/src/pages/AddExpense.jsx` and `frontend/src/pages/AddIncome.jsx` — both amount `<input type="number">` fields now have `inputMode="decimal"` (correct mobile keypad) and `onWheel={(e) => e.target.blur()}` (a scroll over the field while it's focused no longer silently changes the amount).

### 7. Remove hardcoded `$` from the hero amount inputs (A2)
**Status: Done**

- `frontend/src/pages/AddExpense.jsx` and `frontend/src/pages/AddIncome.jsx` — both now pull `symbol` from `useCurrency()` instead of hardcoding `$`. Scoped exactly to these two forms per the bucket's cross-reference; the broader `CurrencyContext` number-formatting fix (thousands separators, negatives) is left for bucket 03.

## Bucket 03 — Frontend: Currency, Dates & Cross-App Polish

Source: `review_order/03-frontend-currency-dates-polish.md`

### 1. UTC date bugs — shared local-date util (A2)
**Status: Done**

- New `frontend/src/utils/dates.js` — `toLocalISODate(date)`, extracted from the correct local-date formatting that already existed in `AddExpense.jsx`. Formats a `Date` as `YYYY-MM-DD` off its local calendar fields (`getFullYear`/`getMonth`/`getDate`), never via `toISOString()` (which converts to UTC first and can silently roll the date backward or forward a day).
- `frontend/src/utils/analytics.js` — `getDateRange()` now builds `start_date`/`end_date` with `toLocalISODate()` instead of `.toISOString().split("T")[0]`.
- `frontend/src/pages/Analytics.jsx` — two more spots doing the same thing wrong, both fixed:
  - The **YTD** preset's `end_date` (was `now.toISOString().split("T")[0]`).
  - The **month drill-down** navigation from the Spending Velocity chart to History: `end_date: new Date(y, m, 0).toISOString().split("T")[0]` — this is the exact bug described in the review ("clicking July filters to July 1–30 in UTC+ zones"), since local midnight of the month's last day converts to the *previous* day in UTC.
- `frontend/src/pages/AddExpense.jsx` and `frontend/src/pages/AddIncome.jsx` — both had their own inline copy of the same local-date logic for computing `today`; both now call the shared `toLocalISODate()` instead.
- Covered by new `frontend/src/utils/dates.test.js`, which pins `process.env.TZ` to `Asia/Kolkata` (UTC+5:30) to reproduce the bug directly: an early-morning local instant that `toISOString()` would roll back to the previous UTC day, the month-drill-down last-day-of-month case, and a sanity check that the bug condition genuinely exists for the test's chosen instant before asserting the fix.

### 2. Currency formatting via `Intl.NumberFormat` (A2)
**Status: Done**

- New `frontend/src/utils/currency.js` — `formatCurrency(amount, symbol)` using `Intl.NumberFormat` for thousands separators, with the negative sign placed before the symbol (`-$12.00`, not the old `$-12.00`).
- `frontend/src/UserPreferencesContext.jsx` (see item 3 — this is where `CurrencyContext`'s logic now lives) uses `formatCurrency` for its `fmt()` output instead of the old `` `${symbol}${amount.toFixed(2)}` ``.
- Covered by new `frontend/src/utils/currency.test.js` (thousands separators, multi-level separators, negative sign placement with both single- and multi-character symbols, `-0` not rendering as `-$0.00`, string-coerced and non-numeric input).

### 3. Per-device → per-user preferences: income-mode and currency (A2)
**Status: Done**

- **Backend** (`backend/models.py`, `backend/database.py`, `backend/main.py`): `UserPreference` gains `currency` (default `"CAD"`) and `income_mode_enabled` (default `False`), alongside the existing `date_format`. Since `SQLModel.metadata.create_all()` never alters an already-existing table (per CLAUDE.md), a new `database.ensure_user_preference_columns()` runs an idempotent `ALTER TABLE ... ADD COLUMN` for any already-deployed database that only has the old `date_format`-only shape, called from the app's startup `lifespan` right after `create_db_and_tables()`. `GET /api/user-preferences` now returns all three fields; `PUT /api/user-preferences` accepts any subset of them (all now `Optional`) so existing partial-update callers (e.g. just `{date_format: ...}`) keep working, and validates `currency` against a new `VALID_CURRENCIES` set the same way `date_format` is validated.
- **Frontend**: `CurrencyContext.jsx`, `DateFormatContext.jsx`, and `hooks/useIncomeMode.js` are now thin re-export shims over a single new `frontend/src/UserPreferencesContext.jsx`, which fetches `/api/user-preferences` **once** and serves all three preferences from one provider (mounted in `App.jsx` as `DateFormatProvider`, post-login) — avoiding three separate components each independently GET-ing the same database row. All ~20 existing `useCurrency()`/`useDateFormat()`/`useIncomeMode()` call sites are unchanged.
  - **Upgrade path for existing users:** on first load, if the server value for currency/income-mode is still at its default *and* the old per-device `localStorage` key holds a different, previously-chosen value, that value is pushed to the server once and the legacy `localStorage` key is cleared — so upgrading doesn't silently reset a couple's currency selection or income-tracking toggle back to the default.
  - `Navbar.jsx`'s currency `<select>` and `Settings.jsx`'s date-format buttons and income-tracking toggle now wrap their preference-setter calls in `try/catch` with a toast on failure — these are new network calls where they used to be instant, synchronous `localStorage` writes that couldn't fail.
- New backend tests in `backend/tests/test_user_preferences.py` (defaults for all three fields, currency update + persistence, invalid currency rejected with 422, partial updates don't reset the other fields, per-user isolation extended to currency) and a new `backend/tests/test_preference_column_migration.py` exercising `ensure_user_preference_columns()` directly against a hand-built "legacy" SQLite table missing the new columns (columns get added with the right defaults, existing data is untouched, running it twice is a no-op, and it's a no-op against a database with no `userpreference` table yet).

### 4. Dark-mode contrast on Spending Velocity (A2)
**Status: Done**

No new Tailwind plugin needed — `MergeDescriptionsModal.jsx`'s local toast implementation (which referenced `animate-in fade-in`, classes from a plugin that was never installed, so they silently did nothing) was replaced with the shared `useToast()` instead of adding the dependency.
  - Wired up: `MergeDescriptionsModal.jsx` (merge/dismiss/restore confirmations), `AddExpense.jsx` and `AddIncome.jsx` (save confirmations), `History.jsx` (delete confirmation, export failure), `Settings.jsx` (avatar upload, app-mode change, stay-signed-in toggle), `Navbar.jsx` (currency change failure).

## Bucket 04 — Backend: Calculation Correctness & Reconciliation

Source: `review_order/04-backend-calculation-correctness.md`

### 1. Payment/balance semantics (B1)
**Status: Done**

- `backend/routes/expenses.py` — `_validate_expense` now rejects any `Payment`-category expense whose `split_method` isn't `"100% <name>"`, with the message *"Payment must use a 100% split to the person being paid"*. Combined with the existing "can't be 100% to the payer themselves" check, this fully forces a Payment into `100% <recipient>` — the only split that means "settle the debt in full." A `Payment` with `50/50` or `Personal` (which previously moved the balance by the wrong amount, e.g. only half the settlement) is now rejected on both create and edit.
- Existing test `test_accept_all_standard_categories` (`backend/tests/test_expenses.py`) was creating its Payment case with the default `50/50` split — updated to use a `100%` split, since that's now the only valid one.
- New tests: `test_payment_rejects_50_50_split`, `test_payment_rejects_personal_split`, `test_payment_accepts_100_percent_split`, `test_payment_still_rejects_100_percent_self`, `test_payment_update_rejects_50_50_split`.

### 2. Reimbursement reconciliation — one convention everywhere (B1)
**Status: Done**

- Adopted the review's suggested convention, now documented in `CLAUDE.md` ("Money & currency") and as a code comment at the top of `backend/routes/analytics.py`: **Reimbursement nets into every totals figure; it's excluded only from `distribution`** (the percentage-of-whole view, since a refund can't sensibly own a negative slice of a pie chart). `Payment` stays excluded everywhere — it's a balance settlement, not spend.
- `backend/routes/analytics.py` — `by_category` (the bar chart) and `by_payer` previously excluded Reimbursement; both now include it, netting refunds into their respective totals. `distribution` was split into its own query (`dist_rows`) so it keeps excluding Reimbursement while `by_category` no longer shares that query. `total_spend`, `total_shared_spend`, `by_split_method`, `over_time`, and `my_share` were already correct (no change).
- `backend/routes/expenses.py` — `get_monthly_summary` no longer excludes Reimbursement; it now appears as its own (typically negative) category line, and the display filter changed from "amount > 0" to "amount != 0" so that negative line isn't silently dropped. This is what makes the category list sum to the same figure as `my-expense-summary`'s `my_total` for the same month — previously they diverged by exactly the reimbursed amount. `get_personal_summary` had the same fix (dropped its Reimbursement exclusion) for the same reason, even though it wasn't named explicitly in the review's bullet list — it's the same class of bug and it's compared against the same per-user monthly totals.
- `top_expenses` (largest single transactions) intentionally still excludes Reimbursement — it isn't one of the "totals" being reconciled, and a negative refund wouldn't rank as a "largest expense" regardless.
- Updated tests to match the new convention: `test_analytics_reimbursement_excluded_from_distribution` (split into a distribution-only assertion), `test_analytics_reimbursement_included_in_by_category` (new), `test_analytics_reimbursement_nets_into_payer_breakdown` (new, replaces the old "excluded" test), `test_monthly_summary_nets_reimbursement` (`test_balance.py`, replaces the old "excludes" test), `test_personal_summary_nets_reimbursement` (`test_modes.py`, new).

### 2. Backups: cadence, verification, and the ignored integrity check (Exec #4, B3)
**Status: Done**

- `backend/database.py` — `check_db_integrity()` now catches the case where the file isn't parseable as SQLite at all (sqlite3 raises `DatabaseError` rather than returning an error row from `PRAGMA integrity_check` for a badly-enough-corrupted or non-SQLite file) and treats it as a failed check instead of letting the exception escape uncontrolled.
- `backend/main.py` (`lifespan`) — the integrity check's result is no longer discarded: if it fails, the app now **refuses to start** with a `RuntimeError` naming the DB path and the backup directory to restore from, and the startup backup step is skipped entirely (a corrupt DB no longer gets backed up over a good prior backup).

  - **Verification**: new `BackupManager.verify_backup(dest)`, called automatically after every `create_backup()`. A failed verification is logged as an error (the backup is kept for forensics rather than silently deleted) rather than assumed to have succeeded. ### 3. Migration scripts silently corrupt data (Exec #5, B3)
**Status: Done — behavior change, please read before next use**

- `backend/migrate_expenses.py` and `backend/migrate_income.py` — both rewritten around the review's four fixes:
  - **`--date-format` is now a required CLI argument** (`choices`: `YYYY-MM-DD`, `MM/DD/YYYY`, `DD/MM/YYYY`), refusing to guess. Previously both scripts tried `%m/%d/%Y` before `%d/%m/%Y` for any string date cell, so a DD/MM sheet (the app's own default display format) silently imported day/month-swapped for every row where day ≤ 12, and only errored (skip + warning) on the rest — one import produced self-inconsistent history with no way to tell which rows were affected. This is the point of the fix (a partial import with silently-dropped rows is worse than refusing to run), but it means re-running an old command line without `--date-format` will now fail immediately rather than guessing.

### 5. Sessions are non-sliding (B4)
**Status: Done — sliding expiry (the smaller-behavioral-change option, per the bucket's own recommendation)**

### 3. Odd-cent regression pack (B2.1) — highest protection
**Status: Done — and found a real, live money-precision bug in the process**

- New `backend/tests/test_portion_consistency.py` — `test_my_portion_expr_and_my_portion_agree_for_odd_cents` exercises every valid `(split_method, paid_by)` combination (Personal/50-50/100%-each-direction) across 8 odd-cent amounts, comparing `_my_portion_expr`'s SQL result against `_my_portion`'s Python result for the same row.
- `test_reconciliation.py::test_odd_cent_split_reconciles_across_balance_share_summary_and_insights` — a single $100.01 50/50 expense, checked across `/api/balance`, `/api/monthly-summary`, `/api/my-expense-summary`, and `/api/insights`' weekend/weekday breakdown, per the bucket's exact ask.
- **This test caught a real bug**, not a test artifact: `/api/balance` reported the payer's half as `$50.01`, while `/api/my-expense-summary`/`/api/monthly-summary` (both built on `_my_portion_expr`) reported `$50.00` for the *identical* expense. Root cause: `_my_portion_expr`'s SQL `CASE` expression has a bare integer literal `0` as its *first* branch's `THEN` value (the `category == "Payment"` guard) — SQLAlchemy infers a `CASE` expression's overall result type from that first branch, so instead of inferring `Numeric(10, 2)` (matching `Expense.amount`'s real column type, which is what `get_balance`'s equivalent `CASE` — whose first branch is itself a Numeric division — correctly infers), it fell back to a generic/untyped result. Aggregate `SUM()`s over that expression came back as bare Python floats instead of scale-aware `Decimal`s, silently losing the DB-level rounding that `get_balance` was getting for free, and the two paths diverged by a cent right at the half-cent boundary.
  - **Fix**: `routes/expenses.py`'s `_my_portion_expr` now wraps its `CASE` in an explicit `cast(..., Numeric(10, 2))`, so its result type is always correct regardless of branch ordering. Confirmed root cause and fix empirically (reproduced the raw-float-vs-Decimal divergence directly against the SQLite engine, then confirmed the cast eliminates it) before applying it — see the `_my_portion_expr` docstring for the full explanation, left in place so a future refactor doesn't reorder the branches back into the same trap.
  - This is a real, silent, live discrepancy: any user with an odd-cent 50/50 (or 100%) split near a half-cent boundary would see their balance and their "my spend this month" figures disagree by a cent. It's exactly the class of bug this bucket's item 3 exists to catch.

### 2. Recurring detection — period grid (C1)
**Status: Done**

- `backend/routes/insights.py` — `_classify_frequency`'s five fixed bands (dead zones at 19–24, 36–79, 101–339 days) are replaced by `PERIOD_GRID` (`{7, 14, 30.44, 60.9, 91.3, 182.6, 365.25}` days → Weekly/Biweekly/Monthly/Bimonthly/Quarterly/Semiannual/Annual) and `_classify_period`, which scores every grid period by the fraction of a series' intervals matching `k × period` (±20%) for the best-fitting small integer `k` per interval, and picks the period reaching ≥60% conformance. **Tie-break subtlety found and handled**: a run of 30-day gaps also "fits" a 14-day (biweekly) period at k=2, or a 7-day (weekly) period at k=4, with equally perfect ±20% conformance — naively picking "whichever period scores highest" is ambiguous. `_score_period` also returns the average `k` used across matched intervals, and `_classify_period` tie-breaks in favor of the *lowest* average k (the more natural, less-coincidental explanation of the same data) — verified this correctly resolves Monthly-vs-Biweekly-vs-Weekly and Bimonthly-vs-Monthly and Semiannual-vs-Quarterly ambiguities in the new tests below.

### 3. Forecast — scheduled + variable (C3, includes the Exec #1 fix)
**Status: Done — full rebuild, not a patch**

`_forecast` is rebuilt around `forecast = scheduled + variable` instead of a category-level weighted moving average:

- **Scheduled component**: for each detected recurring series (from item 2) not flagged `"overdue"`, new `_project_due_dates(next_due, period_days, window_start, window_end)` computes every due date landing in the target month (next calendar month) via closed-form `k_min`/`k_max` arithmetic (not a loop — no risk of a long-overdue series causing a slow or unbounded walk), multiplied by `median_amount`/`median_my_amount` (median of the series' last 3 occurrences). An annual/quarterly bill now appears **only** in its actual due month — verified directly (see below) — instead of being smeared across every month or missed by the old per-category classification.
- **"Is this recurring" is now a per-series question, not per-category**: the old bug where one recurring merchant made its *entire category* "recurring" is gone — `recurring_member_ids` (from item 2) excludes exactly the expenses belonging to a detected series from the variable computation, expense-by-expense, regardless of category.
- **Variable component**: per category, **median** of the last 3 *complete calendar months* (`_complete_month_keys`, anchored to real month boundaries, not the old rolling-window-anchored-to-last-expense-date) of spend **minus recurring-series amounts**.
- **Seasonality**: when ≥13 months of history exist (`SEASONALITY_MIN_MONTHS`) and a category's last-3-months spend is high-variance (coefficient of variation > 0.5, `_seasonal_estimate`), the median is blended 70/30 with the same month last year (`SEASONALITY_BLEND_WEIGHT`). Applied separately for the next-month headline forecast and the current-month "on pace" baseline (each blends against *its own* same-month-last-year, not the other's). `seasonality_applied` (whether 13+ months of history exist at all) is surfaced in the payload.
- **"On pace" for the current month** (new `on_pace` object) — MTD actual (real postings so far this month) + remaining scheduled bills (due dates still ahead in the current month) + daily variable rate × days left. This is the number `review_order/09-...md` earmarks for the Landing page.
- **Range, not a point estimate** (new `range` field) — P25/P75 of the last up to 6 complete-month totals (`statistics.quantiles`), `null` when fewer than 2 months of history exist rather than a misleadingly narrow/wide guess.
- **Subscription roll-up / upcoming bills** (new `recurring_summary` and `upcoming_bills` fields) — active-series count, monthly-equivalent and annual-equivalent cost (each series' median amount converted via `30.44 / period_days`), and the next 10 upcoming due dates sorted ascending. This is the `"12 active recurring ≈ $214/mo ≈ $2,568/yr"` payload the review doc calls out.
- **Exec #1's inverted-weights bug is moot** — there are no more recency-weighted periods at all; `last_month_total`/`change_vs_last_month_pct`/`by_category[].last_month` are now literally the actual sum for the most recently completed calendar month, computed once, not derived from any weighting scheme.
- **Backward-compatible response shape**: `recurring_total`, `variable_total`, `total_forecast`, `shared_*` equivalents, `last_month_total`, `change_vs_last_month_pct`, and `by_category` (now `type: "variable"` only — scheduled bills are reported separately in `scheduled_bills`, never mixed into the same array) all keep their original field names, so `ForecastSection.jsx` needed no changes. Covered by two new cases in `insightsPreview.test.js`.

### 2. Price-increase alerts — step detection, not last-vs-average (C2)
**Status: Done — full rebuild**

- `backend/routes/insights.py` — `_detect_recurring_alerts` (mean-vs-last, fired repeatedly while the mean caught up, went silent once it did, never said what happened) is replaced by `_detect_price_step_alerts`, doing genuine change-point detection per series:
  - `_trailing_run_length` finds how many trailing occurrences are within 5% of the latest amount — this is "how many occurrences confirm the current price."

### 4. Anomalies — robust statistics (C5)
**Status: Done**

- `backend/routes/insights.py` — `_detect_anomalies` replaces mean/stdev z-scores with the **modified z-score** (`0.6745 × (x − median) / MAD`), threshold 3.5, computed **leave-one-out** (the candidate's own amount is excluded from the baseline it's compared against, matched by expense id so two different expenses can share an amount without one masking the other). "Low" anomalies (an unusually cheap grocery run) are dropped entirely — only `z_score > threshold` (not `abs(z_score)`) is ever checked, so `direction` is now always `"high"`.
  - **Correctness fix found while implementing, not just satisfying a spec line**: when the leave-one-out baseline has zero natural variance (MAD = 0 — e.g. ten identical $50 grocery runs), dividing by zero needed handling. Using `float('inf')` for "any deviation is infinitely anomalous" would have been mathematically clean but is a real, live bug waiting to happen: Python's `json.dumps` serializes `float('inf')` as the bare token `Infinity`, which is **not valid JSON** — the frontend's `response.json()` would throw and the whole Insights page would break. Fixed with a small floor (1% of the median, minimum 1 cent) instead, keeping the score large but finite.
  - `category_mean`/`category_std` are renamed `category_median`/`category_mad` (they're no longer mean/stdev, and reporting them under the old names would be actively misleading) — `frontend/src/components/insights/AnomaliesSection.jsx`'s one reference to `category_mean` updated accordingly (and its copy changed from "avg" to "usual"); `z_score`'s field name is unchanged (still populated, just computed differently) so no other frontend reference needed updating.

  - **Worth flagging honestly**: I initially tried to also prove leave-one-out changes the flagged/not-flagged *verdict* (a classic "candidate inflates its own baseline" masking scenario), the way it dramatically would for mean/stdev. For median/MAD specifically this isn't achievable with a realistic minority-outlier dataset — median/MAD has a 50% breakdown point, so a small number of outliers can't shift the median or MAD away from the normal cluster regardless of leave-one-out. Leave-one-out for this statistic is still correct and worth keeping (it avoids biasing the *reported* baseline figures, and is what the review explicitly asks for), but its practical benefit here is precision of the displayed numbers, not preventing missed detections the way it would for a mean-based approach.

### 1. Custom category entry was rejected with a 422

**Status: Done**

- Root cause: the frontend already fully supported custom categories (the "+ New Category" option, free-text input, and submit path in `frontend/src/pages/AddExpense.jsx`), but `backend/routes/expenses.py`'s `_validate_expense` rejected anything not in the hardcoded `VALID_CATEGORIES` whitelist with `422 "invalid category"`.
- `backend/routes/expenses.py` — the whitelist check is replaced with a non-empty check plus a **case-insensitive collision guard**: a new category is rejected only if it differs solely by case from an existing one (e.g. `"groceries"` vs `"Groceries"`, which would otherwise fragment analytics into two buckets) — this also protects the two reserved categories, `Payment` and `Reimbursement`, whose exact spelling drives money math (forced 100% split, the negative-amount exemption, exclusion from balance/analytics) from being silently imitated by a look-alike custom category (e.g. `"payment"`). Genuinely novel names pass through untouched. The stored category is also stripped of surrounding whitespace on create/update.
- `backend/migrate_expenses.py` — the CSV/XLSX importer's category check was loosened to match (same case-collision guard, reusing `_VALID_CATEGORIES_CASEFOLD` from `routes/expenses.py` rather than duplicating it). Fixed a latent related bug found while testing this: openpyxl returns `None` (not `""`) for a truly empty category cell, and `str(None)` is the non-empty string `"None"` — under the old code this at least surfaced as an "unknown category" error, but under the loosened rule it would have silently imported as a valid custom category named `"None"`. The cell is now checked for `None` explicitly before stringifying.
- `frontend/src/utils/validation.js` — added the same case-insensitive collision check against `CATEGORIES` for a custom category, so the user sees `"'<Name>' already exists — select it from the category list."` inline before submitting, instead of only finding out after a round-trip 422.

### 2. Itemized split calculator for uneven/mixed receipts

**Status: Done**

- Idea behind this: for a mixed receipt (some items are only one person's, some are shared, some are taxable), the only existing way to log it was to compute the net owed offline and enter it manually as a "100% other owes" expense. The calculator does that computation in-app. Per the product owner's decisions: itemized (not a plain arithmetic calculator or a flat percentage split), with a taxable toggle and a custom tax rate per bill, launched from a button in the Add Expense form (not a global floating icon), and the result is stored as a `100% <ower> owes` expense — **no schema change**, matching the existing manual workaround exactly.
- New `frontend/src/utils/splitCalc.js` — pure function `computeSplit(items, taxPct, payer, userA, userB)`. Each item carries an `owner` (one of the two display names, or the literal `"Shared"`) and a `taxable` flag. An item owned by the payer contributes nothing to what's owed (it was always theirs), an item owned by the other person is owed in full (plus tax if taxable), and a Shared item is owed at half. Returns `{ subtotal, tax, total, netOwed, ower }`, rounded to 2 decimal places to match the server's `Decimal` quantization.
- New `frontend/src/components/SplitCalculatorModal.jsx` — modeled on the existing `MergeDescriptionsModal.jsx` overlay pattern (backdrop, centered card, Escape-to-close, click-outside-to-close). Lets the user pick who paid, set a tax % (defaults to 13%, remembered in `localStorage` across sessions), add/remove line items with a Mine/Theirs/Shared toggle and a per-item taxable checkbox, and shows a live subtotal/tax/total/"X owes Y" preview. "Use this result" is disabled whenever the computed net owed is 0 (nothing to log) so it can't produce the zero-amount expense the backend rejects.
- `frontend/src/pages/AddExpense.jsx` — added a "Split calculator" button under the amount field (shared/blended mode only — hidden entirely in Personal mode, where there's no one to owe). On "Use this result" the form's `amount`, `paid_by`, and `split_method` are set directly (`amount` = net owed, `paid_by` = whoever paid the bill, `split_method` = `100% <the other person>`), reusing the form's existing state shape — the live split-preview sentence under the amount field updates immediately since it's already reactive to those three fields.
- New `frontend/src/utils/splitCalc.test.js` — covers an empty item list, an item owned by the payer contributing nothing, an item owned by the other person owed in full, a shared item at half, tax applied only to taxable items, a full mixed-bill scenario (shared+taxable, other's non-taxable, payer's own), switching which person paid, ignoring zero/blank-amount items, and rounding to 2 decimal places.
